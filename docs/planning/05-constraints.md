# 제약 조건 및 설계 원칙

## 핵심 설계 원칙

### 1. 폴더트리가 진실의 원천 (Single Source of Truth)
- 폴더트리 JSON이 변경되면 마인드맵, 사이드바, Hero 모두 자동 반영
- 별도로 마인드맵을 편집하는 UI 없음 — 폴더트리만 관리하면 됨
- 폴더 하나를 통째로 넣거나 빼면 모든 뷰가 즉시 갱신

### 2. 마크다운 퍼블리싱, 포스팅이 아님
- 시계열 포스트가 아니라 구조화된 퍼블리싱
- 폴더/파일 구조가 콘텐츠의 계층과 관계를 표현
- 업데이트는 파일 수정으로, "새 글 올리기"가 아님

### 3. CSS 테마로 개성 표현
- 테마 = CSS Variables 세트
- 마크다운 리더 + 마인드맵 + Hero가 동시에 테마 반영
- 방문자도 테마 전환 가능 (읽기 편의)

## 기술적 제약

| 항목 | 제약 |
|------|------|
| 서버 용량 | Supabase 무료 티어 (500MB DB, 1GB Storage) |
| 배포 | Vercel 무료 티어 |
| 인증 | MVP에서는 관리자(소유자) 1명만 로그인, 방문자는 익명 |
| 댓글 | 로그인 없이 닉네임만 입력 (스팸 위험 있으나 MVP에서 감수) |
| 모바일 | 반응형 필수 — 폴더트리는 드로어 방식 |

## 비즈니스 제약

| 항목 | 제약 |
|------|------|
| 타임라인 | MVP 챌린지 발표까지 (빠른 프로토타이핑 필요) |
| 사용자 규모 | 초기 20명 (MVP 챌린지 멤버) |
| 수익 모델 | 없음 (MVP 단계) |

## 하지 않을 것 (Explicitly Out of Scope)

- 옵시디언 플러그인 개발
- 다른 사용자 URL 자동 크롤링 (2차 MVP에서)
- DM 기능 (홀드)
- 피드/타임라인 (2차 MVP에서)
- 결제/구독 시스템
